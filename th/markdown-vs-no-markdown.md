# มี Markdown vs ไม่มี Markdown

เปรียบเทียบผลกระทบเมื่อโปรเจค **มี** เอกสาร markdown ที่ดี กับเมื่อ **ไม่มี**

---

## 1. สำหรับนักพัฒนา (Developer)

| ด้าน | ✅ มี Markdown | ❌ ไม่มี Markdown |
|------|---------------|-----------------|
| การเริ่มงาน | คนใหม่อ่าน README แล้วเริ่มได้ในไม่กี่นาที | ต้องถามเพื่อนร่วมทีม เสียเวลาหลายชั่วโมง |
| การ Setup | มีขั้นตอนชัดเจนใน CONTRIBUTING.md | "รันยังไง?" ถามใน Slack ทุกสัปดาห์ |
| สถาปัตยกรรม | มีเอกสารใน docs/ เข้าใจง่าย | ความรู้อยู่ในหัวคน ไม่มีที่บันทึก |
| Changelog | CHANGELOG.md บอกว่าเปลี่ยนอะไร ทำไม | `git log` ยาวเหยียด ไม่มีบริบท |
| Code Review | PR template ช่วยให้ review สม่ำเสมอ | PR ไม่สม่ำเสมอ ขาดบริบท |
| ประวัติการตัดสินใจ | ADR อธิบายว่าทำไมถึงเลือกแบบนี้ | "ทำไมเราถึงเลือก X?" — ไม่มีใครจำได้ |
| API | มีเอกสาร endpoint, request/response | อ่านซอร์สโค้ดแล้วเดาเอา |
| การ Deploy | มี runbook ทีละขั้นตอน | "ถาม DevOps" หรือหวังว่า CI จะทำงาน |

### ผลกระทบจริง

```
ไม่มี markdown:
  คนใหม่เข้ามา → 2 สัปดาห์กว่าจะทำงานได้
  เจอ bug → 1 ชั่วโมงกว่าจะเข้าใจระบบ
  Deploy → "ใครรู้วิธี deploy บ้าง?"

มี markdown:
  คนใหม่เข้ามา → 2 วันก็ทำงานได้
  เจอ bug → 10 นาทีก็หา module ที่ถูกต้องได้
  Deploy → ทำตาม runbook
```

## 2. สำหรับผู้ใช้ AI Agent

| ด้าน | ✅ มี Markdown | ❌ ไม่มี Markdown |
|------|---------------|-----------------|
| บริบท | Agent อ่าน specs, steering, rules → ผลลัพธ์แม่นยำ | Agent เดาจากโค้ดอย่างเดียว → ผลลัพธ์ทั่วไป |
| สไตล์โค้ด | Agent ทำตาม convention ที่บันทึกไว้ | Agent ใช้ค่าเริ่มต้นของตัวเอง ไม่ตรงกับโปรเจค |
| สถาปัตยกรรม | Agent เข้าใจขอบเขตและ pattern | Agent อาจละเมิดสถาปัตยกรรม (ผิด layer, ผิด pattern) |
| ความถูกต้อง | Agent รู้กฎเฉพาะโปรเจค (เช่น "ราคาเก็บเป็นสตางค์") | Agent สมมติเอง ทำให้เกิด bug |
| ประสิทธิภาพ | prompt เดียว → ผลลัพธ์ถูกต้อง | ต้องแก้หลายรอบ |
| ความปลอดภัย | Agent รู้ว่าอะไรห้ามแตะ | Agent อาจแก้ไฟล์สำคัญโดยไม่รู้ตัว |

### ผลกระทบจริง

```
ไม่มี markdown:
  Prompt: "เพิ่ม payment endpoint"
  ผลลัพธ์: CRUD ทั่วไป, pattern ผิด, convention DB ผิด
  เวลาแก้: 30+ นาที

มี markdown:
  Prompt: "เพิ่ม payment endpoint"
  Agent อ่าน: AGENTS.md, steering rules, pattern ที่มีอยู่
  ผลลัพธ์: สถาปัตยกรรมถูก, ตั้งชื่อถูก, error handling ครบ, มี test
  เวลาแก้: 0–5 นาที
```

## สรุป

```
┌─────────────────────────────────────────────────────────┐
│           โปรเจคที่ไม่มี MARKDOWN                         │
├─────────────────────────────────────────────────────────┤
│  Developer: "ระบบนี้ทำงานยังไง?"                          │
│  AI Agent:  "ผมจะเดาจากโค้ดที่เห็น"                       │
│  ผลลัพธ์:   ช้า, ไม่สม่ำเสมอ, ผิดพลาดบ่อย                 │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│           โปรเจคที่มี MARKDOWN                            │
├─────────────────────────────────────────────────────────┤
│  Developer: *อ่านเอกสาร* "เข้าใจแล้ว ทำตามนี้"            │
│  AI Agent:  *อ่าน specs* "ผมจะทำตามกฎเหล่านี้"            │
│  ผลลัพธ์:   เร็ว, สม่ำเสมอ, เชื่อถือได้                    │
└─────────────────────────────────────────────────────────┘
```

---

## บทสรุป

ไฟล์ Markdown ไม่ใช่แค่เอกสาร — มันคือ **บริบทที่ใช้งานได้จริง** สำหรับทั้งมนุษย์และ AI การใช้เวลา 30 นาทีเขียน README ที่ดี ช่วยประหยัดเวลาหลายชั่วโมงทุกสัปดาห์ สำหรับทุกคน (และทุก agent) ที่ทำงานกับโปรเจค
